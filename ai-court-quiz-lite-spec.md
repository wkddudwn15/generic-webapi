# AI裁判クイズ Lite 仕様書

## 概要

「AI裁判クイズ Lite」は、AIが生成した事件ファイルを読み、3つの証言の中から矛盾しているものを選ぶシンプルな推理クイズです。

裁判ゲームの雰囲気は残しつつ、操作は「読む」「選ぶ」「答えを見る」だけにします。現在のリポジトリにあるクイズ生成アプリの構造を活かしやすく、`prompt.md` と `public/quiz.html` を置き換えるだけで実装できます。

## コンセプト

- プレイヤーは陪審員または見習い裁判官
- AIが短い事件、証拠、3つの証言を作る
- 3つの証言のうち、1つだけ事件内容や証拠と矛盾している
- プレイヤーは矛盾している証言を選ぶ
- 正解後、AIが「どこが矛盾していたか」を説明する

## 基本ループ

1. プレイヤーがテーマと問題数を選ぶ
2. フロントエンドが `/api/` にテーマと問題数を送る
3. LLMが事件クイズをJSONで返す
4. 画面に事件ファイル、証拠、3つの証言を表示する
5. プレイヤーが矛盾している証言を選ぶ
6. 正誤と解説を表示する
7. 次の事件へ進む
8. 最後にスコアを表示する

## ゲームの目的

各事件で、もっともらしい3つの証言から「1つだけおかしい証言」を見抜くことです。

## MVP仕様

- 問題数を選べる
- テーマを選べる
- 事件タイトルを表示する
- 事件概要を表示する
- 証拠を2から3個表示する
- 証言を3つ表示する
- プレイヤーは証言を1つ選ぶ
- 正解、不正解を表示する
- 解説を表示する
- 最終スコアを表示する
- リスタートできる

## 難易度

### やさしい

- 矛盾が分かりやすい
- 証拠と証言を比べればすぐ分かる
- 文章は短め

### ふつう

- 1段階だけ推理が必要
- 時刻、場所、持ち物、発言内容などを照合する

### むずかしい

- 直接の矛盾ではなく、複数の情報を組み合わせる必要がある
- ただし理不尽なひっかけにはしない

## 1問の構成

1問は以下の要素を持ちます。

- `caseTitle`: 事件タイトル
- `caseSummary`: 事件概要
- `evidence`: 証拠リスト
- `testimonies`: 証言3つ
- `answer`: 矛盾している証言のインデックス
- `explanation`: 正解理由
- `judgeComment`: 裁判官の短いコメント

## APIリクエスト例

```json
{
  "title": "AI裁判クイズ Lite",
  "count": 5,
  "theme": "オフィス",
  "difficulty": "ふつう"
}
```

## APIレスポンス例

現在の `server.js` は、LLMレスポンスのトップレベルオブジェクト内から最初の配列を取り出します。そのため、LLMは必ず `{ "data": [...] }` の形で返します。

```json
{
  "data": [
    {
      "caseTitle": "消えた会議室の鍵",
      "caseSummary": "午後3時、会議室Aの鍵がなくなった。鍵は昼休み前まで受付の机に置かれていた。",
      "evidence": [
        "受付カメラには、12時10分に佐藤が受付前を通る姿が映っている。",
        "会議室Aは13時から14時まで予約されていなかった。",
        "鍵は15時20分に給湯室の棚から見つかった。"
      ],
      "testimonies": [
        "佐藤: 昼休み中はずっと外にいました。受付には近づいていません。",
        "田中: 13時半ごろ、給湯室で佐藤さんを見ました。",
        "鈴木: 15時すぎに鍵がないことに気づきました。"
      ],
      "answer": 0,
      "explanation": "証拠では12時10分に佐藤が受付前を通ったとされているため、「受付には近づいていない」という証言と矛盾します。",
      "judgeComment": "証拠映像は強力です。証言の信用性に疑いがあります。"
    }
  ]
}
```

## UI方針

- 裁判風だが、ルール説明を大量に出さない
- プレイヤーが最初に見る操作は「証言を選ぶ」だけにする
- 証拠と証言は読み比べやすく配置する
- 1問あたりの文章量は短めにする
- 正解後の解説は1から2文にする

## 判定方式

正誤判定はフロントエンドで行います。

LLMは問題生成だけを担当します。プレイヤーが選んだ証言のインデックスと、問題データ内の `answer` を比較して正誤を判定します。

この方式にすると、毎回答え合わせでAPIを呼ばなくてよいため、ゲームが軽くなります。

## 実装メモ

- 現在の `MAX_COUNT` は20なので、問題数は1から20の範囲で使えます。
- `server.js` は起動時に `prompt.md` を読み込むため、プロンプトを変更したらサーバー再起動が必要です。
- `extractArray` はトップレベルオブジェクト内の最初の配列を取り出すため、LLMレスポンスは必ず `{ "data": [...] }` にします。
- 既存のクイズアプリと同じく、`answer` は0始まりのインデックスにします。
