# しりとり裁判 仕様書

## 概要

「しりとり裁判」は、しりとりの回答をAI裁判官が審理する単語ゲームです。

プレイヤーが入力した単語に対して、AIが「有効」「却下」「敗訴」などの判決を出し、判定理由を証拠として表示します。通常のしりとりに、ジャンル縛り、異議申し立て、ライフ、スコアを加えることで、会話型LLM APIの強みを活かしたゲーム体験にします。

## コンセプト

- 普通のしりとりを、裁判の演出でゲーム化する
- AIを「対戦相手」ではなく「裁判官兼対戦相手」として使う
- 単語の読み、ジャンル適合、使用済み判定などをLLMに審理させる
- 判定理由を短く表示し、納得感のあるゲームにする
- 不服がある場合は「異議あり」で再審理できる

## 基本ループ

1. プレイヤーが単語を入力する
2. フロントエンドが現在の状態と単語を `/api/` に送信する
3. `server.js` が `prompt.md` に変数を埋め込み、LLMへ送信する
4. LLMがJSONで判決を返す
5. フロントエンドが判決、証拠、AIの返し手を表示する
6. 有効なら次の指定文字を更新する
7. 無効ならライフを減らす
8. ライフが0になるか、敗北条件を満たしたらゲーム終了

## プレイヤー状態

- `lives`: ライフ。初期値は3
- `objections`: 異議チケット。初期値は2
- `score`: スコア
- `combo`: 連続有効回答数
- `currentKana`: 次に始めるべき文字
- `usedWords`: 使用済み単語の一覧
- `turns`: ターン数
- `gameOver`: ゲーム終了状態

## ゲームモード

### 通常裁判

通常のしりとりです。日本語の名詞として自然で、前の単語の最後の文字から始まっていれば有効です。

### 専門法廷

ジャンル縛りつきのしりとりです。

例:

- 食べ物
- IT用語
- 地名
- 動物
- スポーツ
- 学校にあるもの

### 厳格法廷

判定を厳しめにします。

- 一般名詞を優先する
- 固有名詞は原則無効
- 俗語や造語は慎重に扱う
- ジャンル一致を厳密に見る
- 読みが曖昧な単語は却下しやすくする

## 難易度

### ゆるい

- 読みや表記に多少の揺れがあっても有効にする
- カタカナ語や略語を広く認める
- ジャンル一致も広めに解釈する

### 標準

- 一般的なしりとりとして自然な判定をする
- 名詞として自然な単語を有効にする
- ジャンル縛りは常識的に判定する

### 厳格

- 名詞性、読み、ジャンル一致を厳しく見る
- 固有名詞、略称、俗語、曖昧な言葉は却下しやすくする

## 判定ルール

AI裁判官は、各ターンで以下を確認します。

- `userWord` が空ではない
- `userWord` が日本語の単語として自然である
- `userWord` が名詞として扱える
- `userWord` の読みが `currentKana` から始まる
- `userWord` が「ん」で終わっていない
- `userWord` が `usedWords` に含まれていない
- ジャンル指定がある場合、ジャンルに合っている
- 小さい文字、長音、濁点、半濁点を自然なしりとりルールとして処理する

## 文字処理ルール

LLMには次の方針で判定させます。

- ひらがな、カタカナ、漢字混じりを読みとして解釈する
- 最後の文字が長音 `ー` の場合、直前の母音または実際の読みを考慮する
- 最後の文字が小さい `ゃ`, `ゅ`, `ょ`, `っ` の場合、自然な読みで次文字を決める
- 濁点・半濁点は基本的に区別してよいが、難易度が「ゆるい」場合は柔軟に扱う
- 「を」は通常「お」として扱ってよい

## 判決タイプ

### `valid`

プレイヤーの単語が有効です。AIの返し手も表示し、次の指定文字を更新します。

### `invalid`

プレイヤーの単語が無効です。ライフを1減らします。

### `lose`

プレイヤーが即敗北です。主に「ん」で終わった場合に使います。

### `ai_lose`

AIが返し手を出せない、またはAIが敗北条件を満たした場合です。プレイヤー勝利です。

### `retrial`

異議申し立てによって再審理した結果です。最終的には `valid` または `invalid` 相当の扱いをします。

## 裁判演出

### 起訴

プレイヤーの単語に疑いがある場合、短い争点を表示します。

例:

> 起訴内容: 指定文字と異なる読みで始まっている疑いがあります。

### 証拠

判定理由を1から2文で表示します。

例:

> 「りんご」は「り」から始まる食べ物で、まだ使用されていません。

### 判決

画面上では次のように表示します。

- 有効: `判決: 有効`
- 無効: `判決: 却下`
- 敗北: `判決: 敗訴`
- AI敗北: `判決: 勝訴`
- 再審理: `判決: 再審理`

## 異議あり

プレイヤーは無効判定に対して、1ゲームに2回まで異議を申し立てられます。

異議申し立て時は、前回の単語、前回の判決、プレイヤーの主張をLLMに送ります。

例:

```json
{
  "appealReason": "サーバはIT用語として一般的です。長音なし表記も使われます。"
}
```

AIは再審理し、判定を維持するか覆します。

## APIリクエスト例

```json
{
  "title": "しりとり裁判",
  "mode": "専門法廷",
  "genre": "食べ物",
  "difficulty": "標準",
  "currentKana": "り",
  "userWord": "りんご",
  "usedWords": "しりとり",
  "lives": 3,
  "objections": 2,
  "score": 0,
  "combo": 0,
  "appealReason": ""
}
```

## APIレスポンス例

`server.js` はLLMレスポンス内の最初の配列を `data` として返すため、LLMはトップレベルに `data` 配列を持つJSONを返します。

```json
{
  "data": [
    {
      "verdict": "valid",
      "charge": "問題なし",
      "evidence": "「りんご」は「り」から始まる食べ物で、まだ使用されていません。",
      "userWord": "りんご",
      "userReading": "りんご",
      "aiWord": "ごま",
      "aiReading": "ごま",
      "nextKana": "ま",
      "message": "次は「ま」から始まる単語を提出してください。",
      "gameOver": false,
      "penalty": 0,
      "scoreDelta": 10
    }
  ]
}
```

## フロントエンド画面構成

### ヘッダー

- アプリ名
- モード
- ジャンル
- 難易度

### ステータスバー

- ライフ
- 異議チケット
- スコア
- コンボ
- 現在の指定文字

### 中央エリア

- AI裁判官の判決カード
- 起訴内容
- 証拠
- AIの返し手
- 次の指定文字

### 入力エリア

- 単語入力欄
- 提出ボタン
- 異議ありボタン
- リスタートボタン

### 履歴エリア

- 使用済み単語一覧
- プレイヤーの単語とAIの単語をターンごとに表示

## スコア仕様

- 有効回答: `+10`
- ジャンル一致が明確: `+5`
- コンボボーナス: `combo * 2`
- 異議で判定が覆った場合: `+5`
- 無効回答: `0`

スコア計算はMVPではフロントエンド側で行い、LLMからの `scoreDelta` は参考値として扱ってもよいです。

## 実装対象ファイル

- `prompt.md`
  - しりとり裁判の判定プロンプトに変更する
- `public/quiz.html`
  - しりとり裁判の画面に置き換える
- `public/quiz.css`
  - ゲーム用スタイルに変更する
- `public/style.css`
  - 共通レイアウトを必要に応じて調整する
- `server.js`
  - MVPでは原則変更しない

## 注意点

- 現在の `server.js` は `prompt.md` を起動時に読み込むため、`prompt.md` を変更したらサーバー再起動が必要です。
- 現在の `extractArray` は、LLMレスポンス内の最初の配列を取り出します。そのため、レスポンスは必ず `{ "data": [...] }` 形式にします。
- 複数アプリを同時に動かす構成ではないため、クイズはしりとり裁判に置き換える前提です。
- LLM判定には揺れがあるため、判定理由を表示し、異議あり機能でゲームとして吸収します。
